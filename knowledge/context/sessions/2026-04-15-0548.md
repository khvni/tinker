## Decisions made

- **Layout version bump strategy**: on a version mismatch we silently log and fall back to the default layout instead of attempting auto-migration. This matches the spec's "safer to fall back to default than to attempt auto-migration" guidance.
- **Debounce window**: 300 ms. Short enough to survive an app crash mid-drag; long enough that a rapid split-drag sequence writes once. Reused the vault reindex constant's ballpark.
- **Agent-path resolution**: if OpenCode reports an absolute path we open it as-is; relative paths are resolved against the active vault root and dropped silently if no vault is attached (matches the "app remains usable when integrations disconnected" invariant).
- **Auto-open toggle**: spec mentions a "open created files automatically" preference (default on). Deferred — not required for acceptance criteria and would need Settings UI work. Currently auto-open is always on when a vault is attached. Flagging in open questions below.
- **Code renderer**: still `<pre><code>`-based. CodeMirror 6 vs Monaco decision deferred. `getCodeLanguage` still computes the right `language-*` class for a future syntax-highlighting renderer swap.

## Open questions

- **Auto-open preference toggle**: should this land with feature 07 or wait until there's a Settings home for workspace-level preferences? Current behavior: always on when vault is attached. Feels right as a default; toggle can follow when Settings grows a section for workspace behavior.
- **Code renderer upgrade**: still open from the feature spec. Suggest tracking as a follow-up issue once actual code tabs get heavier real-world use — the current `<pre>` renderer is functional for small files.
- **Multi-session chat tabs**: not implemented. Only one `chat` pane in the default layout, and session ID is held on a ref inside `Chat`. Opening two Chat panes today would share the same `sessionIDRef` unless the component re-renders fresh — needs follow-up. Not an acceptance-criteria blocker for feature 07.

## Gaps surfaced

- `packages/bridge/src/stream.ts` has no tests; the `file_written` event path is now load-bearing for auto-open. A test that asserts patch + file parts both emit `file_written` would harden this.
- No end-to-end test proves the Dockview round-trip (save → relaunch → restore). Would need Playwright + a Tauri dev build. Worth considering once the FirstRun flow stabilizes.
- `Workspace.openFileInWorkspace` is not covered by unit tests — it's tightly coupled to a Dockview API instance. A thin harness that mocks `DockviewApi` could cover the "existing panel reuse vs new panel" branches.
