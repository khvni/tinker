## Decisions Made

- **Git via Tauri `Command`, not `simple-git` / `@napi-rs/simple-git`.** Reason: `simple-git` runs in Node (not the webview); `@napi-rs/simple-git` bundles native binaries and inflates the app. Shelling out inherits the user's credential helper and keeps the bundle lean. Trade-off: requires git on PATH — surfaced via `isGitAvailable()` and a notice in the Dojo Git pane. Documented in the feature spec.
- **Working tree at `<vault>/.tinker/`** (not `<vault>/.tinker/skills/`) so tracked files land at `skills/*.md` in the remote — matches the spec's remote layout convention and keeps the repo isolated from any vault-level Git the user might have.
- **Prompt-injection skills only for v1** — sub-agent primitives deferred to feature 06. `injectActiveSkills` sends a single `noReply` text part at session creation, not every turn — avoids ballooning context. Toggles bump `activeSkillsRevision`, Chat aborts the active session, and the next prompt spins up a fresh session with the refreshed injection.
- **Activation state in the `skills` table.** A separate `active_skills` table was considered but rejected — one column keeps joins trivial and the app has no multi-user scope.
- **Dojo accessed via a default Dockview tab** (next to Today/Settings). The spec mentioned a command palette; no palette exists yet, so adding one was explicitly out of scope for this feature. Pane auto-registers when a vault is connected.

## Open Questions / Handoff

- **Git credentials UX**: the system `git` uses whatever credential helper is configured. If a user hits an SSH passphrase / 2FA prompt during `syncSkills`, the Command call hangs (no stdin piping). Surface a timeout? Or instruct users to set up credential helpers first? Leaving for review-round feedback.
- **Sub-agent activation (feature spec "Open Questions")**: still deferred. When feature 06 lands, revisit whether some skills should register as sub-agents instead of prompt injection. Current frontmatter already accepts `tools: […]`; we can extend with `type: subagent` later without a schema change.
- **Skill ID collision on Git pull** (spec open question): not yet solved. A skill pulled from the remote with the same slug as a locally authored one will overwrite via merge. For v1 this is acceptable (team convention); add a namespacing scheme (e.g., `<remote>/<slug>`) when this hurts.
- **Sensei scoring** lives in feature 05, as noted in the spec. The SQLite FTS index is ready for it to query.

## Gaps Surfaced

- **OpenCode SDK `noReply` prompt**: confirmed via existing `memory-injector.ts` pattern — worked fine. No SDK reading required this session.
