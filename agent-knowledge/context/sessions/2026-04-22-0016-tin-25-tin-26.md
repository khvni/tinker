# TIN-25 + TIN-26 — chat history JSONL persistence + hydration

Contributor: khvni
Agent: Codex

## Scope
- Deliver `TIN-25` and `TIN-26` together as one PR for [[21-mvp-session-folder]] and [[28-mvp-identity]].
- Persist per-user OpenCode session events to JSONL in the session folder and hydrate Chat from that file before streaming resumes.

## Changes
- Added `packages/bridge/src/chat-history.ts` with:
  - queued append-only JSONL writer (`createChatHistoryWriter`)
  - JSONL reader / parser with malformed-tail recovery
  - latest-history discovery in `.tinker/chats/<user-id>/`
- Extended `packages/bridge/src/stream.ts` so Chat can attach raw-event side effects to the same filtered SSE subscription it already uses for UI streaming.
- Added bridge coverage in `packages/bridge/src/chat-history.test.ts` for queued writes, mkdir-on-first-write, error logging, JSONL parsing, and latest-session discovery.
- Added `findLatestSessionForFolder` to `packages/memory/src/session-store.ts` plus tests, so Chat can reopen the most recent session for the active folder/user pair.
- Added `apps/desktop/src/renderer/panes/Chat/historyReplay.ts` + tests to replay stored OpenCode events back into `ChatMessage`/`MessageBlock` state without needing OpenCode history as the source of truth.
- Wired `Chat.tsx` to:
  - resolve current folder/user session on mount
  - fall back to latest JSONL file when SQLite is missing
  - restore a missing session row from chat history
  - block composer input during hydration
  - append raw SSE events to JSONL while the existing UI stream continues
- Wired `Workspace.tsx` to pass current user id + current folder path into Chat runtime.
- Added `@tauri-apps/plugin-fs` to `@tinker/bridge` and refreshed `pnpm-lock.yaml`.

## Decisions Made
- Reused the existing Chat SSE subscription instead of opening a second event stream; JSONL writes now run as a side effect on the same filtered event flow the UI already consumes.
- Used exact-line dedupe on JSONL hydration rather than the spec note's `(event, id)` sketch, because `message.part.delta` legitimately emits many entries with the same `partID`.
- Kept scope inside current pane/session architecture. No pane-schema expansion for per-pane `sessionId` landed here.
