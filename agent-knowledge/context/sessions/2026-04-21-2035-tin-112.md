# TIN-112 — Playbook skill schema + frontmatter spec

## Outcome

Defined a canonical Playbook skill schema in `@tinker/shared-types`, taught `packages/memory/src/skill-parser.ts` to normalize legacy markdown into that shape, and shipped the slice to review in PR #46.

## What shipped

- `packages/shared-types/src/skill.ts` now exports `DEFAULT_SKILL_VERSION`, a canonical `SkillFrontmatter` (`id`, `title`, `role`, `tools`, `version`, `author`) and a normalized `SkillDefinition`/`Skill` shape that keeps `slug` as a legacy alias.
- `packages/memory/src/skill-parser.ts` now reads both canonical and legacy frontmatter (`id` or old `name`), normalizes to canonical frontmatter on serialize, and defaults missing versions to `1.0.0`.
- `packages/memory/src/skill-service.ts` now hydrates persisted skills with `id`, `role`, `version`, and `author` so the richer type is available after reindex/load/install flows.
- `apps/desktop/src/renderer/panes/Playbook.tsx` now preserves `title`, `role`, `version`, and `author` when editing an installed skill, avoiding metadata loss even though the deferred UI still only exposes the legacy fields.
- Coverage added in `packages/shared-types/src/skill.test-d.ts`, `packages/memory/src/skill-parser.test.ts`, and `packages/bridge/src/skill-injector.test.ts`.

## Review notes

- Stayed inside ticket scope: types, parser normalization, persistence hydration, and one compatibility fix in the deferred Playbook pane.
- Deliberately did not add new Playbook UI fields or broader marketplace/runtime behavior; D25 still defers the feature beyond this schema slice.
