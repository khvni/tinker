## Discriminator shape rationale

Considered three variants:

1. `TinkerPaneData` as kind-less payload (`{ path, mime }` for file, empty for others) — lost discrimination at the data level, forced call sites to cross-reference `pane.kind` separately.
2. `TinkerPaneData` as `{ kind, data }` wrapper — nested `data.kind` + `data.data.path`; noisy narrowing.
3. **Chosen**: `TinkerPaneData` as flat discriminated union with `kind` as the tag and kind-specific fields inline (`{ kind: 'file'; path; mime }`). Narrowing works via `if (data.kind === 'file')`; `Pane<TinkerPaneData>` stores `pane.data: TinkerPaneData` where the outer `pane.kind` string mirrors `data.kind` by registry convention.

The chosen shape matches task M1.4's literal registration pattern `{ kind: 'file', data: { path, mime } }` when interpreted as "the pane registration has `kind: 'file'` and its data carries the `file`-variant of `TinkerPaneData`".
