# Session — Sign-in UX (M8.9)

## Scope

Dedicated full-window sign-in route that gates app boot behind a Better Auth session. Subdivided per the spec note (>500 LOC threshold) into three pieces:

- **Provider picker** — three secondary `Button`s (Google / GitHub / Microsoft) with per-provider error annotations.
- **Waiting-for-browser** — pulse `StatusDot`, `Cancel`, and a 60 s "Open browser again" affordance.
- **`useAuthSignIn` hook** — minimal state machine `idle | signing-in` with cancel-token guard.

## What was built

### `apps/desktop/src/renderer/routes/SignIn/`

- `SignIn.tsx` — shell that picks `ProviderPicker` vs `WaitingForBrowser` based on hook state.
- `SignIn.css`, `SignIn.test.tsx`, `index.ts`.
- `hooks/useAuthSignIn.ts` — initiates `onSignIn(provider)`, drives state, runs the 60 s retry timer; cancel bumps `cancelTokenRef` so the eventual resolve from the deferred Rust call is ignored.
- `hooks/useAuthSignIn.test.ts` — pure-helper tests (`IDLE_PHASE`, `phaseForStart`, `phaseAfterRetryDelay`) per the project's no-jsdom test pattern.
- `components/ProviderPicker/` — folder-per-component, uses `Button` with `variant="secondary"` (no internal overrides).
- `components/WaitingForBrowser/` — folder-per-component.

### `apps/desktop/src/renderer/App.tsx`

Inserted a sign-in gate that early-returns `<SignIn>` when `nativeRuntime && !hasSignedIn`. `Workspace` now also requires `hasSignedIn` — boot blocks until the user signs in. The existing `handleProviderConnect` is reused unchanged (it already does `upsertUser` + `getActiveMemoryPath`); the new SignIn route just delegates to it via the `onSignIn` prop. Errors flow through the existing `providerMessages` state (single source of truth) and are forwarded to `ProviderPicker` as a prop — no error tracking duplicated inside the hook.

### `apps/desktop/src/renderer/routes/design-system.tsx`

Added a `'sign-in'` tab with two stories (idle, runtime-unavailable). The in-flight story uses `() => new Promise<void>(() => {})` so it parks in `signing-in` indefinitely without leaking a 250 s `setTimeout`.

### `agent-knowledge/context/tasks.md`

M8.9 row flipped from `not started` → `review`, with PR #79 reference.

## Decisions

- **Cancel is renderer-only.** The Rust `auth_sign_in` command runs to completion (up to 180 s) regardless of what the UI does; we just stop awaiting. Documented inline with a `cancel-token` ref pattern that bumps on cancel and gates `setPhase` on the awaited resolution. Acceptable because (a) the next sign-in spawns a fresh ticket, (b) the abandoned task self-times-out.
- **No new design primitives.** First Executor pass tried to stuff a leading glyph + two-line label into the existing `Button`, fighting its fixed 26 px height and single-row layout. Reviewer flagged. Round 2 dropped the glyph + subtitle and uses `Button` as designed; subtitle copy moved into the picker's header paragraph.
- **Errors live in `App.tsx`, not the hook.** Round-1 hook had its own `errors: ProviderErrorMap` + `state='error'` branch that was unreachable in production because `handleProviderConnect` never re-throws (it `setProviderMessage`s and swallows). Round 2 deleted the parallel state and threads `providerMessages` straight through.
- **No jsdom / no `@testing-library/react`.** Followed the project pattern (cf. `useSessionHistoryWindow.test.ts`, `draftReducer.test.ts`): components rendered with `renderToStaticMarkup`, hooks decomposed into pure-function seams that are unit-tested directly. The brief originally suggested `renderHook`, but introducing two devDeps for one feature ran counter to CLAUDE.md "make complexity invisible, not absent."
