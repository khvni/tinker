# TIN-53 — M5.1 OpenCode SDK usage + context window research

## Outcome
Added `agent-knowledge/reference/opencode-sdk-usage.md` documenting the exact SDK field paths the context-usage badge (M5) will read from. Unblocks M5.2 (badge UI) and M5.3 (wiring).

## Key findings
- **No session-aggregate token field exists.** `Session` (`v2/gen/types.gen.d.ts:766`) carries no `tokens`. The badge must compute from `AssistantMessage.tokens` (authoritative, `types.gen.d.ts:488`) — `{ total?, input, output, reasoning, cache: { read, write } }`. Prefer `tokens.total` when present; otherwise `input + output + reasoning`.
- **Current-window vs lifetime.** For the badge use the **last** `AssistantMessage.tokens` (what sits in the model's window going into next turn), not a running sum. Running sum ≠ window occupancy once history slides.
- **Model context window** lives at `client.config.providers()` → `providers[providerID].models[modelID].limit.context` (`Model.limit`, `types.gen.d.ts:1459`). Cache per process boot; invalidate on `installation.updated` / `server.connected`.
- **Live stream:** subscribe to `event.subscribe()` → `message.updated` (final tokens) + `message.part.updated` for `step-finish` parts (intra-turn increments).
- **Reset signals:** `session.compacted` event rewrites history — badge must reset in-flight counter and re-fetch `session.messages()`. `ContextOverflowError` in `AssistantMessage.error` union pins badge to 100%.
