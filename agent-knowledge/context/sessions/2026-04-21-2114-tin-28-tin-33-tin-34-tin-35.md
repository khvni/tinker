# TIN-28 + TIN-33 + TIN-34 + TIN-35 — FilePane dispatch + HTML/code/markdown preview hardening

## Outcome

Moved the combined M3.2 / M3.7 / M3.8 / M3.9 slice to review in PR #59.

## What landed

- Threaded `{ path, mime }` through the `FilePane` Dockview adapter so existing renderers can use MIME-aware behavior without adding new dispatch layers.
- Kept unsupported-file fallback in `FilePane`, but extracted the shell-open call into a testable helper and added coverage for the OS handoff path.
- Locked `HtmlRenderer` to `sandbox="allow-same-origin"` and added a conservative "Open externally for full interactivity" hint plus `console.warn` when the source HTML contains `<script>` tags.
- Extended `getCodeLanguage()` to prefer specific MIME matches while still falling back to extension-based detection for generic `text/plain`.
- Reworked `MarkdownRenderer` so GFM stays enabled and code fences reuse the shared `code-highlighter` path via Marked `walkTokens`, avoiding a duplicate markdown-only highlighter.
- Added focused tests for FilePane fallback, HTML preview posture, MIME-based code-language detection, markdown GFM rendering, and existing highlighter regression coverage.

## Verification

- `pnpm --filter @tinker/desktop exec vitest run src/renderer/panes/FilePane/FilePane.test.tsx src/renderer/renderers/file-utils.test.ts src/renderer/renderers/HtmlRenderer.test.ts src/renderer/renderers/MarkdownRenderer.test.ts src/renderer/renderers/code-highlighter.test.ts`
- `pnpm --filter @tinker/desktop typecheck`
- `pnpm -r typecheck && pnpm -r test`

## Notes

- The HTML "full interactivity" hint is intentionally heuristic: presence of `<script>` tags is used as the trigger. That catches the important unsafe/JS-backed preview case without trying to simulate runtime interactivity inside the sandbox.
- `MarkdownRenderer` still carries the legacy Dockview edit-button path because TIN-35 only covered preview dispatch + rendering parity, not editor migration.
