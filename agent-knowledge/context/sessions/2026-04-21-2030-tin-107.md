# TIN-107 — Enterprise SSO: SAML provider adapter (docs-only)

**Parent:** TIN-92 (Enterprise SSO milestone — enterprise-fork only).

## What shipped

New doc `docs/saml-adapter.md` covering:

1. **Why SAML is architecturally different** from the existing Entra OBO / Okta XAA path — assertions are XML artifacts, not bearer tokens; forks must **replace** the Better Auth sidecar rather than layer a `FederationAdapter`.
2. **Library recommendation:** `@node-saml/node-saml` (TS, active maintenance, MIT, signature + assertion validation built in). Alternatives table with posture notes on `saml2-js`, `samlify`, `passport-saml`.
3. **Architectural overview + ASCII sequence diagram** showing the fork keeps upstream's public sidecar HTTP surface (`/desktop/sign-in/:provider`, `/desktop/finish`, `/desktop/session`, `/health`) byte-for-byte compatible while swapping SAML in behind it.
4. **9-step fork recipe** — IdP registration, SP key generation, sidecar rewrite with illustrative TypeScript, attribute → `IdentitySession` mapping, assertion validation checklist (8 mandatory checks), key storage rules, `opencode.json` shape, renderer impact (zero), federation/integration consequences.
5. **Loopback-vs-hosted SP tradeoffs** for IdPs that reject `127.0.0.1` ACS URLs.
6. **Negative test matrix** (expired, wrong audience, wrong signature, replay, no `InResponseTo`).
7. **Upstream upgrade guidance** — keep all SAML-specific code concentrated in `main.ts` + new `src/saml.ts` so upstream merges become one-file resolutions.
8. **FAQ** — SCIM scope, group/role attributes, SLO, encrypted assertions, metadata generation, mixed SAML + OIDC deployments.

Linked from `docs/README.md`, `docs/enterprise-fork-guide.md` (new SAML alternative section sitting alongside Okta XAA / Google Workspace), `docs/auth-architecture.md` (FederationAdapter implementations table + Related links), and `agent-knowledge/features/01-sso-connector-layer.md` Out of Scope entry.

## Key design calls made

- **Sidecar swap, not adapter.** SAML doesn't produce exchangeable bearer tokens, so no `SAMLFederationAdapter` shape works. Forks keep the `IdentitySession` contract and replace the Better Auth bits wholesale. Entire renderer + downstream code path unchanged.
- **`canFederate()` returns false for all services on a SAML fork.** Integrations fall through to per-service OAuth. If orgs want true silent federation across apps, they need an OIDC layer in front of their SAML IdP and use the existing Entra OBO / Okta XAA path instead. Stated explicitly so forks don't try to synthesize downstream tokens from SAML assertions.
- **Assertion validation checklist is mandatory, not optional.** Eight explicit items (signature coverage, audience, destination, time window, subject confirmation, replay protection, IdP-initiated disabled, algorithm allowlist). Framed as an audit list forks run before shipping.
- **Loopback ACS preferred, hosted-SP fallback documented** with an explicit "do not" on terminating SAML at a shared SaaS gateway (breaks audience restriction, voids trust).
- **No SCIM content.** Out of scope; SCIM is orthogonal to SAML + Tinker has no central user directory to provision into. Punted to forks that add one.
