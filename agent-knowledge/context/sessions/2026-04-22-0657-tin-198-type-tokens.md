# TIN-198 — Typography parity vs Paper 9J-0

## What shipped

- **Role tokens** layered on top of existing generic scale (`--font-size-xs`..`--font-size-xxl`).
  - `display` (28/700/1.2), `h2` (20/600/1.3), `h3` (16/500/1.4), `body` (14/400/1.45), `secondary` (13/400/1.4), `label` (11/500/1.3 + uppercase + 0.08em), `mono` (12/400/1.4).
- **Dark override**: `--font-weight-display` flips from 700 → 600 under `[data-theme="dark"]` per Paper 6M-0. Only theme-dependent type weight in the system.
- **New generic step**: `--font-size-xxs: 10px` for pixel-fit edge cases (Avatar 20 px circle, KeyboardHint wide key). Justified because two consumers exist; inline value would break the no-literal-px guard.
- **Hardcode sweep**: Avatar, KeyboardHint, SelectFolderButton, panes tab/close glyphs, WorkspaceSidebar avatar, design-system playground, panes-demo eyebrow, code-content line-height all switched to token references.
- **Playground**: `"Type Roles (Paper 9J-0)"` tab demos all 7 roles with exact Paper sample strings + a scoped dark preview verifying the weight flip.
- **Guard script**: `script/check-typography-tokens.sh` greps for `font-size: <N>px` in `packages/` and `apps/desktop/`. Exits 1 on drift.

## Decisions

- Generic scale stays load-bearing for primitives that don't map 1:1 to a Paper role (e.g. `--font-size-sm` used inside panes fallbacks). Roles are additive, not a replacement.
- Pixel-aligned `line-height` values kept as literal `px` with comments (SelectFolderButton pill, WorkspaceSidebar avatar) because they exist to align a single element to a fixed pixel box; tokenizing would obscure intent.
- Dark display-weight override is intentional and matches Paper 6M-0 exactly. Without it, display-sized headings would need hand-coded per-theme blocks, violating D14 tokens-only discipline.
