# TIN-110 — Enterprise SSO: Entra tenant-lock pattern

**PR**: https://github.com/khvni/tinker/pull/39 (In Review)
**Branch**: `khvni/tin-110`
**Linear**: https://linear.app/tinker/issue/TIN-110
**Scope**: post-mvp, enterprise-fork-only docs (per D1 / D8 / D25)

## What the ticket asked for

> Fork-guide section: swap Microsoft consumer endpoint `common` for specific tenant ID.
> Enforce `tid` claim equality on assertion.

Two tenant-lock invariants that an enterprise fork of Tinker must apply on top of upstream's Better Auth Microsoft provider. Upstream targets consumer users — it points Microsoft OAuth at `/common/…` so personal, work, and school accounts all work. An enterprise fork that leaves `common` in place lets users from any Microsoft directory complete sign-in against the fork's app registration, which defeats tenant isolation.

## What landed

Docs-only edits to `docs/enterprise-fork-guide.md`:

1. **Step 3 callout** — new blockquote explaining that `common` must be replaced by the tenant UUID on every Microsoft endpoint (authorize, token, OBO, JWKS, discovery). Points forks at a concrete grep pattern (`login.microsoftonline.com/common`) so the check is automatable during fork CI.
2. **Step 6 prose + adapter code** — introduces `decodeJwtClaims` + `assertTenantLocked` helpers inside the `createEntraOBOAdapter` example. The helpers:
   - decode the incoming ID token JWT (Node `Buffer` base64url round-trip — the auth sidecar runs on Node so no Web Crypto gymnastics needed)
   - fail closed on missing `tid` / `iss` / `aud` / `exp`
   - reject if `tid !== config.tenant`, if `iss !== https://login.microsoftonline.com/<tenant>/v2.0`, if `aud` is neither the client ID nor `api://<clientId>`, or if the token is expired
   - run BEFORE the OBO exchange fetch, so the foreign-directory token never touches Entra
   Prose frames this as belt-and-suspenders: the endpoint lock alone is usually sufficient, but the assertion check defends against config drift, a dev proxy, or a rotated tenant UUID that hasn't propagated everywhere yet.
3. **Step 7 checklist** — adds two verification items (endpoint grep returns zero matches; unit test that a foreign-tenant token throws `Tenant mismatch`).

## Why the scope stopped there

Per D1 / D8 / D25 the enterprise federation *runtime* does not live in upstream. This ticket is explicitly labelled `docs` + `post-mvp`. Implementing `packages/auth-sidecar/src/adapters/entra-obo.ts` for real would violate the fork boundary and pull enterprise compliance surface into upstream. The fork-guide code fence is the canonical place for this pattern.

Elon 5-step pass:

- **Requirements**: the ticket is one specific, load-bearing fork instruction — accepted.
- **Delete**: did not add a new Step; folded the lock into existing Steps 3 / 6 / 7. No parallel "Security" section.
- **Simplify**: no JWT library dependency — raw base64url decode via `Buffer`. The snippet is self-contained enough for a fork to copy without first auditing a new dep.
- **Accelerate / Automate**: the grep-for-`common` check is a one-liner a fork can paste into its CI.
