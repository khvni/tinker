# TIN-87 — M8.14 Chat history JSONL format doc

## Outcome

Wrote `agent-knowledge/reference/chat-history-format.md` as a short spec for `<folder>/.tinker/chats/<user-id>/<session-id>.jsonl`. Doc-only PR, no code.

## What shipped

- Line shape `{ ts: ISO8601, event: string, data: unknown }` defined and pinned to `Event.type` + `Event.properties` from `@opencode-ai/sdk@1.4.3` v2.
- Enumerated every session-scoped OpenCode SSE event type (`session.*`, `message.*`, `message.part.*`, `permission.*`, `question.*`, `todo.updated`) with example JSON payloads.
- Mapped the logical names in the ticket (`user.prompt`, `assistant.delta`, `tool.use.start`, `tool.use.end`, `message.thinking.delta`, `assistant.complete`, `error`, `abort`) onto their concrete SDK events so M2.11/M2.12 implementers don't guess.
- Implementation notes cover append semantics, crash safety (best-effort non-blocking writes, tolerant truncated-tail reads), concurrency (one writer per `(user, session)`), hydration ordering (read JSONL first, subscribe after), and rotation policy (none in MVP).

## Why this shape

- `{ ts, event, data }` keeps the file forward-compatible: unknown event types flow through unchanged when the SDK adds new ones.
- Filtering to `properties.sessionID === activeSessionID` cleanly excludes global/LSP/TUI/VCS noise without the writer having to know specific event types.
- Hydration sequencing was captured explicitly because feature 28 flagged it as a race risk.
