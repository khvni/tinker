# TIN-7 — M1.5 settings + memory pane placeholders

## Scope

- Deliver [[20-mvp-panes-workspace]] task 1.5: register `settings` and `memory` placeholder panes in the typed `PaneRegistry`.
- Keep current Dockview workspace behavior unchanged until M1.7 swaps the runtime to `@tinker/panes`.

## What shipped

- Added shared `EmptyPane` component under `apps/desktop/src/renderer/workspace/components/EmptyPane/` with tokens-only styling and a friendly empty-state card.
- Added `SettingsPane` and `MemoryPane` component folders that currently wrap `EmptyPane` with pane-specific copy.
- Added `apps/desktop/src/renderer/workspace/register-pane-renderers.tsx` as the single boot entrypoint for pane registration; today it registers `settings` and `memory`, and later M1.3/M1.4 can extend the same file for `chat` and `file`.
- Added server-render tests covering `EmptyPane`, `SettingsPane`, `MemoryPane`, and the registration path.

## Decisions / rationale

- Did **not** wire the new registration entrypoint into the current app boot path. `Workspace.tsx` still renders through Dockview, so invoking the new registry now would add side effects with no runtime consumer. The registration surface is ready for M1.7 without perturbing legacy behavior.
- Used per-kind wrapper components instead of inline registry lambdas so future settings/memory work can replace component internals without changing registration call sites.
