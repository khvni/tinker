# TIN-17 — M2.3 Tauri command `open_folder_picker`

## What shipped

A thin Tauri v2 command that opens the native OS folder picker and returns the selected absolute path. Used by the upcoming first-run + "New session" flows (M2.1 / M2.4) where every Tinker session binds to a local folder.

## Files

- `apps/desktop/src-tauri/src/commands/dialog.rs` (new) — `open_folder_picker` sync command over `tauri-plugin-dialog` v2 `blocking_pick_folder()`
- `apps/desktop/src-tauri/src/commands/mod.rs` — exports `dialog`
- `apps/desktop/src-tauri/src/lib.rs` — registers the command in `invoke_handler!`
- `apps/desktop/src-tauri/permissions/allow-folder-picker.toml` (new)
- `apps/desktop/src-tauri/capabilities/default.json` — grants `allow-folder-picker` to the main window
- `apps/desktop/src/bindings.ts` — adds `openFolderPicker(): Promise<string>` wrapper over `@tauri-apps/api/core` `invoke`

## Notable decisions

- **Sync command, not async.** Tauri v2 dispatches non-async commands on its blocking thread pool, so `blocking_pick_folder()` is safe. First pass used `tokio::sync::oneshot` + the callback `pick_folder`; reviewer (elon-reviewer agent) flagged the oneshot and the extra `sync` tokio feature as adapter code wrapping an API that already exposes the blocking form — deleted both.
- **No `resolve_absolute` helper.** First pass also included a helper that would join a relative path onto `cwd`. Native folder dialogs on macOS/Windows/Linux always return absolute paths, so the fallback branch was dead defensive code. Deleted per Elon step 2 (DELETE) and CLAUDE.md §2 (trust internal guarantees, only validate at system boundaries — the OS dialog *is* the boundary, and it gives an absolute path).
- **Error surface.** Two error strings: `"Folder selection was cancelled."` when the user dismisses the picker, and `"Picked folder is not a local filesystem path: {error}"` for the unreachable-on-desktop `FilePath::Url` case the plugin-fs enum forces us to handle.

## Out of scope / follow-ups

- **Wiring into the renderer.** `openFolderPicker()` is exposed but unused on this branch. The first-run + "New session" surfaces land with TIN-15 (`Session` type) and TIN-18 (`start_opencode`).
- **Session-switch race handling.** Feature spec [[21-mvp-session-folder]] calls out cancelling an in-flight spawn when the user clicks a second session. That's TIN-18's concern, not TIN-17's.
